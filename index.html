Git使用学习笔记2

发表于 2020-03-01 分类于 Git Valine：

什么是分支

在git中每次commit都是把信息保存到分支中，例如前面提到的master分支。
一开始，master是一条直线，git使用master指向最新的提交，再用HEAD指向master确定当前分支，以及当前分支的提交点，每次提交，master都会前进一步。如下图(示意图均来自廖雪峰git教程)

当我们创建新的分支时，例如dev分支，git后新建一个指针dev，指向master的相同提交，如果切换到dev分支，再把HEAD指向dev。

从现在开始，在dev分支的提交不在对master做修改，而是dev每次前进一步，同时HEAD指向dev。

如果要把master合并，只需要把master指向dev就可以了。

此时删除分支也没有什么影响了。

创建分支

git branch dev 创建dev分支
git checkout dev 切换到dev分支
git switch dev 切换到dev分支
git checkout -b dev 创建并切换到dev分支
git switch -c dev 创建并切换到dev分支
尽量使用switch。

然后按照正常使用方式使用就可以了。
如何合并分支呢？

git checkout master 切换回master分支
git merge dev 把dev分支合并到master
git branch -d dev 删除dev分支

##### 总结 #### 解决冲突 如果在f1分支修改后提交，且master也修改后提交，此时会发生冲突。 解决冲突的办法时修改冲突内容为需要的内容，然后再次commit，此时master会超前一步，也就是说分支已经没有用了，可以把master合并到f1,再在f1修改，也可以直接删除f1，重建分支。 用git log --graph命令可以看到分支合并图。 ##### 分支管理策略 在合并分支时，默认使用快速合并，在历史中看不出来怎么合并的 如果禁止快速合并，git会保存合并历史 使用git merge --no-ff -m "str" dev 这样在git log graph 查看历史时会看到合并轨迹，否则不显示合并轨迹。 ##### 保护现场 ##### 删除分支 如果分支已经合并 git branch -d dev 删除分支 如果分支没有合并 git branch -D dev 强制删除分支 ##### 添加标签 由于commit id太难记，我们可以给版本起一个别名，就是标签。 git tag v1.0 给当前HEAD最近一次commit 加标签为v1.0 git tag v0.9 "fjjf55" 给commitid 为fjjf55的版本加标签为v0.9 git tag -a v0.7 -m "说明" 给标签添加说明 git tag 可以查看所有标签 git tag -d v0.7 删除标签 git push origin 推送本地标签 git push origin --tags 推送所有未推送过的本地标签 git push origin :refs/tags/ 删除远程标签

Git使用学习笔记1

发表于 2020-02-29 更新于 2020-03-01 分类于 Git Valine：

什么是git？

git是一个分布式版本控制系统，用于敏捷高效的处理项目。

Git 安装配置

Git 目前支持 Linux/Unix、Solaris、Mac和 Windows 平台上运行。
Git 各平台安装包下载地址为：http://git-scm.com/downloads
安装什么的比较简单，windows下载安装文件一路next就好。linux根据安装工具不同还有不同的安装方式。
且 Git 的工作需要调用 curl，zlib，openssl，expat，libiconv 等库的代码，所以需要先安装这些依赖工具。

Debian/Ubuntu

Centos/RedHat

安装完成后配置自己GitHub信息。

$ git config –global user.name “Your Name”
$ git config –global user.email “email@example.com“

gitHub 的使用

在绑定GitHub账号后可以在本机使用git。

生成ssh

$ ssh-keygen -t rsa -C “email@example.com“
连续按三次回车
cd ~/.ssh 打开id_rsa.pub 复制信息到Github粘贴。

进入需要上传的目录

git init
git add “Filename”
git commit “Str”

上传到Github

到GitHub复制仓库地址
执行指令关联仓库

git remote add origin https://github.com/XXXXX/XXXXX.git
git pull –rebase origin master
git push -u origin master
上传代码

git查看信息

git status 显示有哪些文件被修改
git diff 显示都改了些什么

文件回退

git log 倒序显示提交信息
git reset –hard HEAD^ 文件回退到上一次commit前
git reset –hard HEAD~100 文件回退到100次commit前
git reset HEAD a.txt 将放到暂存区的修改撤销并恢复源文件为暂存区修改。
git checkout – a.txt 将源文件修改恢复到目前master版本。
以上两条连起来就是放到暂存区又不想要如何修改。

如果回退后再次后悔而此时的HEAD已经不再指向”最新”的版本怎么办？
如果终端没有关闭找到要恢复的版本号，就是那一串超级长的数字，输入前几位就可以了。
如果终端关闭了需要根据commitstr 先找到版本号。

工作区 版本库 暂存区

工作区：工作目录
版本库：.git/ 目录
暂存区:.git/stage或.git/index
master：这是我们最终的目标区 其中HEAD指向的就是这里

git add 文件A //把文件A从工作目录暂存至暂存区
git commit //把文件从暂存区上传至master。

更详细的信息可以查看：https://www.liaoxuefeng.com/wiki/896043488029600/897271968352576

如何撤销

git checkout – a.txt

  1. 如果还没有git add(暂存区为空)即已经commit但还没有add，把工作区文件修改全部撤销，恢复到版本库
  2. 1 如果已经git add还没有commit（暂存区非空）把工作区文件恢复为暂存区内容。
  3. 2 恢复为暂存区后可以重新操作也可以git reset –hard HEAD 恢复到版本号。
  4. git reset HEAD 在git add 后与2.1的行为相同都是将源文件恢复到暂存区，不同的是3.会撤销git add 行为即清空暂存区——仅仅针对源文件，其他文件暂存区不变。

删除文件

删除后如果反悔 git checkout – filename
确定删除且需要删除版本库内文件 git rm filename,git commit -m “remove file”

蓝桥杯训练赛——龟兔赛跑

话说这个世界上有各种各样的兔子和乌龟，但是研究发现，所有的兔子和乌龟都有一个共同的特点——喜欢赛跑。于是世界上各个角落都不断在发生着乌龟和兔子的比赛，小华对此很感兴趣，于是决定研究不同兔子和乌龟的赛跑。他发现，兔子虽然跑比乌龟快，但它们有众所周知的毛病——骄傲且懒惰，于是在与乌龟的比赛中，一旦任一秒结束后兔子发现自己领先t米或以上，它们就会停下来休息s秒。对于不同的兔子，t，s的数值是不同的，但是所有的乌龟却是一致——它们不到终点决不停止。
然而有些比赛相当漫长，全程观看会耗费大量时间，而小华发现只要在每场比赛开始后记录下兔子和乌龟的数据——兔子的速度v1（表示每秒兔子能跑v1米），乌龟的速度v2，以及兔子对应的t，s值，以及赛道的长度l——就能预测出比赛的结果。但是小华很懒，不想通过手工计算推测出比赛的结果，于是他找到了你请求帮助，请你写一个程序，对于输入的一场比赛的数据v1，v2，t，s，l，预测该场比赛的结果。
